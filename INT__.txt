As a manual tester what type testing you have done
Manual testers perform a variety of testing types to ensure software quality. Here are some of the types of testing they have done:
Black Box Testing: Focuses on the software's input and output without knowing its internal structure. It's used for functional testing and can be applied to all levels of software testing, including unit, integration, system, and acceptance testing.
1
White Box Testing: Involves knowledge of the internal structure and implementation of the software. It's used for code optimization and finding hidden errors.
1
Gray Box Testing: Combines elements of both black and white box testing, allowing testers to have some knowledge of the internal workings of the software.
1
Acceptance Testing: Conducted by end-users to verify that the software meets their requirements and expectations.
1
System Testing: Ensures the entire software system works correctly together.
1
Integration Testing: Verifies that different software components work together as a unit.
1
Unit Testing: Tests individual units of code to ensure they work correctly independently.
1
Sanity Testing: Ensures the software is free of defects and works correctly under all conditions.
1
Smoke Testing: A quick, informal check to ensure the software is not broken.
1
Regression Testing: Verifies that new code does not break existing features.
1
User Interface (UI) Testing: Ensures the software's UI is functional and user-friendly.
1
Exploratory Testing: A hands-on approach to testing that allows testers to discover defects and explore the software's behavior.
1

Manual testers use these testing types to ensure that the software meets the required quality standards and provides a good user experience.

JMETER OVERVIEW
++++++++++++++++++++++++++++++

1. Test Plan
The root element of any JMeter script.
Defines the overall test configuration, variables, and execution order.
Can include multiple thread groups, logic controllers, and listeners.

2. Thread Group Elements
Thread groups define virtual users and how they execute requests.

Thread Group – Basic group of virtual users; controls number of threads, ramp-up time, and loop count.
Setup Thread Group – Runs before the main test execution (for pre-test setup).
Teardown Thread Group – Runs after the main test execution (for cleanup).
Concurrency Thread Group (from plugins) – Allows precise control over concurrent users.
Arrivals Thread Group (from plugins) – Controls arrival rate of new users.

3. Samplers
Samplers send requests to a server or system under test.

HTTP Request – Sends HTTP/HTTPS requests.
FTP Request – Sends FTP commands.
JDBC Request – Executes SQL queries.
Java Request – Runs custom Java code.
SOAP/XML-RPC Request – Sends SOAP or XML-RPC calls.
JMS Publisher/Subscriber/Point-to-Point – Sends/receives JMS messages.
SMTP Sampler – Sends emails.
TCP Sampler – Sends raw TCP requests.
OS Process Sampler – Executes system commands.
BeanShell Sampler – Runs BeanShell scripts.
JSR223 Sampler – Runs Groovy/Java scripts (preferred over BeanShell).
Debug Sampler – Outputs variable and property values for debugging.

4. Logic Controllers
Control the flow and order of samplers.

Simple Controller – Groups samplers logically.
Loop Controller – Repeats child elements a set number of times.
Once Only Controller – Runs child elements only once per thread.
If Controller – Runs children based on a condition.
While Controller – Loops while a condition is true.
Switch Controller – Runs one child element based on index or name.
Module Controller – References another part of the test plan.
Include Controller – Includes an external test plan file.
Transaction Controller – Measures the time taken for a group of requests.
Throughput Controller – Runs children based on a percentage or throughput.
Runtime Controller – Runs children for a set duration.
Interleave Controller – Alternates between child elements.
Random Controller – Runs children in random order.
Random Order Controller – Executes all children in random order.

5. Config Elements
Provide default settings and variables for samplers.

HTTP Request Defaults
FTP Request Defaults
JDBC Connection Configuration
User Defined Variables
CSV Data Set Config – Reads test data from CSV files.
Counter
Random Variable
Keystore Configuration
Login Config Element
DNS Cache Manager
HTTP Cache Manager
HTTP Cookie Manager
HTTP Header Manager
HTTP Authorization Manager

6. Pre-Processors
Run before a sampler executes.

User Parameters
HTML Link Parser
HTTP URL Re-writing Modifier
BeanShell PreProcessor
JSR223 PreProcessor
Regular Expression Extractor (can also be a post-processor)
XPath Extractor
JSON Extractor
Boundary Extractor

7. Post-Processors
Run after a sampler executes, usually to extract data.

Regular Expression Extractor
XPath Extractor
JSON Extractor
CSS/JQuery Extractor
Result Status Action Handler
Debug PostProcessor
JSR223 PostProcessor
BeanShell PostProcessor

8. Assertions
Validate that responses meet expectations.

Response Assertion
Duration Assertion
Size Assertion
XML Assertion
XML Schema Assertion
HTML Assertion
MD5Hex Assertion
XPath Assertion
JSON Assertion
Compare Assertion

9. Timers
Introduce delays between requests.

Constant Timer
Gaussian Random Timer
Uniform Random Timer
Constant Throughput Timer
Synchronizing Timer
Poisson Random Timer
BeanShell Timer
JSR223 Timer

10. Listeners
Collect and display test results.

View Results Tree
View Results in Table
Summary Report
Aggregate Report
Aggregate Graph
Graph Results
Response Time Graph
Backend Listener – Sends metrics to external systems (e.g., InfluxDB, Grafana).
Simple Data Writer
Mailer Visualizer

11. Non-Test Elements
Utility components not directly part of the test execution.

Test Fragment – Reusable test plan section.
Property Display
Debug Sampler
WorkBench (deprecated) – Temporary storage for test elements.
✅ Tip:
For real-world performance testing, you typically combine:

Thread Groups (load setup)
Samplers (requests)
Config Elements (data/config)
Assertions (validation)
Listeners (reporting)

11)What is Response Time and how is it calculated?
Response time=End time-Start Time
Example:
Request sent at 10:00:01
Response received at 10:00:04
Response Time = 3 seconds

12)What is Average Response Time?
Answer:
It is the mean of all response times collected.
Formula:
Average Response Time = Sum of all response times / Number of samples
Example:
Response times: 2s, 3s, 5s
Average = (2 + 3 + 5) / 3 = 3.33 sec

13)How is Error Percentage calculated?
Answer:
Formula:
Error % = (Number of failed requests / Total requests) × 100
Example:
Failed requests = 120
Total requests = 6,000
Error % = (120 / 6000) × 100 = 2%

8. What is Latency in JMeter?
Answer:
Latency is the time taken for the first byte of the response to be received after sending the request.
Difference:
Latency → Server processing + network delay
Response Time → Latency + data transfer time

How do you calculate Concurrent Users?
Answer:
Concurrent users are users active at the same time.
Formula:
Concurrent Users = Arrival Rate × Average Response Time
Example:
Arrival rate = 5 users/sec
Avg response time = 4 sec
Concurrent Users = 5 × 4 = 20 users

What is Hits Per Second?
Answer:
Hits/sec indicates how many HTTP requests hit the server per second.
Formula:
Hits/sec = Total hits / Test duration (seconds)
Example:
18,000 hits in 300 seconds
Hits/sec = 60
